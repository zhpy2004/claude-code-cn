# Claude Code 汉化工具 — 可行性分析报告

> 日期：2026-04-14  
> 基于 Claude Code v2.1.105，cli.js 13.04 MB

---

## 总评：可行，有条件约束

AST 补丁方案在技术上完全可行，但实验数据揭示了几个需要在设计阶段就解决的关键约束。

---

## 一、UI 字符串可识别性

### 结论：**可行，但需要精准的 AST 上下文过滤**

### 数据

| 分类 | 数量 | 占比 | 说明 |
|------|------|------|------|
| 明确可翻译 | 2,440 | 1.8% | React `createElement` 调用中的文本 |
| 明确不可翻译 | 20,921 | 15.2% | 比较、switch/case、startsWith 等逻辑判断 |
| 需逐个判断 | 113,869 | 83.0% | 函数参数、赋值、数组元素等 |
| **总计** | **137,230** | | |

### 关键发现

1. **createElement 是最可靠的 UI 文本标识符**。2,440 个字符串位于 React `createElement` 调用中，抽样验证全部是用户可见的 UI 文本：
   - "Welcome to Claude Code for"
   - "Press ... again to exit"
   - "Deny, and tell Claude what to do differently"
   - "Summarized conversation"
   - "(No resources found)"

2. **cli.js 是大型 bundle**，混入了大量第三方库字符串（TLS/crypto 错误、DOM 操作、其他语言的本地化文本如瑞典语、印尼语等）。总字符串 159K，但 Claude Code 自身的 UI 文本只是小部分。

3. **UI 关键词密度集中在文件后段**（60%-100% 位置），说明 Claude Code 自身代码在 bundle 后半部分。

### 风险：字符串用途混合

**严重**：某些字符串同时出现在显示和逻辑判断中。例如：
```javascript
T.startsWith("Error: ")  // 逻辑判断，翻译会破坏功能
```
AST 上下文分析显示 `startsWith` 中有 465 个字符串、`===` 比较中有 14,315 个。同一个英文字符串可能在 A 处用于显示、B 处用于比较——翻译 A 处时不能影响 B 处。

### 应对策略

- **P0 阶段只翻译 `createElement` 内的字符串**（2,440 个），这些是最安全、最可靠的 UI 文本
- 使用 AST 精确定位每个字符串的 (start, end) 偏移，逐个替换，不做全局文本替换
- 同一英文文本在不同 AST 位置有不同偏移，可以做到"只翻译显示位置，不翻译逻辑位置"

### 跨平台考虑

| 平台 | 潜在问题 |
|------|----------|
| Windows | `@` 在路径中，需用引号处理；部分安装位置可能需管理员权限 |
| macOS | Homebrew 安装的 Node.js 可能需要 `sudo` |
| Linux | 全局 node_modules 通常在 `/usr/lib/` 下，需要 `sudo` |

### 应对策略

- 路径处理统一用 `path.join()`，兼容 Windows `\` 和 Unix `/`
- 写入前做权限检查，权限不足时给出明确提示（如"请用管理员运行"）

## 四、版本更新影响

### 结论：**高频更新是最大维护挑战**

### 数据

| 指标 | 值 |
|------|-----|
| 总发布版本数 | 374 |
| 最近 30 版平均间隔 | **1.4 天** |
| 最短间隔 | 0.2 天（约 5 小时） |
| 最长间隔 | 4.0 天 |

### 影响分析

Claude Code 平均每 1-2 天发布一个新版本。这意味着：

1. **用户每次 `npm update` 后都需要重新打补丁**
2. **翻译字典需要跟随版本更新**——新版本可能新增/修改/删除 UI 字符串
3. **补丁脚本必须快速且自动化**（当前 ~3.5s，可接受）

### 应对策略

- **npm postinstall hook**：在 `claude-code-cn` 的 postinstall 中检测是否需要重新补丁
- **版本容错**：翻译字典中找不到的新字符串保持英文原文，不影响功能
- **自动检测**：`claude-code-cn status` 命令检查当前版本是否已补丁
- **字典版本无关设计**：字典按英文原文匹配，而非按偏移位置——即使代码重排，只要文本不变就能命中

## 七、对蓝图的修订建议

基于本次可行性分析，建议对原蓝图做以下调整：

### 1. P0 翻译范围缩窄为 `createElement` 内的字符串

原蓝图的 P0 定义为"核心 UI"（数百个），但缺少精确的筛选标准。现在有了明确标准：

- **P0**：`createElement` 调用中的文本字符串（~2,440 个，去重后更少）
- **P1**：其他 AST 上下文中经人工确认的显示文本
- **P2**：模板字面量中的文本片段

### 2. patcher 必须按偏移替换，不能按文本全局替换

同一英文文本可能在不同位置有不同用途。patcher 必须使用 AST 解析后的精确 `(start, end)` 偏移，只替换确认为 UI 文本的那些位置。

### 3. 新增 extractor 的"上下文标签"功能

extractor 提取字符串时，需要附带 AST 上下文标签（`createElement` / `comparison` / `switchCase` 等），用于自动分类和人工审核。

### 4. 翻译字典设计调整

字典不能只是 `{ "english": "中文" }` 的简单映射。需要包含上下文信息：

```json
{
  "entries": [
    {
      "original": "Select model",
      "translation": "选择模型",
      "context": "createElement",
      "file_version": "2.1.105"
    }
  ]
}
```
